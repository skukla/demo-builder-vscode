# Research: AEM Sites as Edge Delivery content source — authentication

**Date:** 2026-06-11
**For:** Commerce-Connect Slice 2 (R1 — AEM content-source auth model)
**Method:** web search + GitHub repo source (Adobe doc pages 403 direct fetch; flagged inline)
**Status:** Verified from sources; 3 items flagged for live confirmation against the available AEM-as-Cloud-Service instance.

---

## Bottom line

When **AEM Sites** is the content source, content is **pushed** from AEM author into the Edge Delivery pipeline on publish; EDS reads back rendered markup from a `…/bin/franklin.delivery/<owner>/<repo>/main` endpoint on the AEM **author** host, authorized by a **Site Authentication Token tied to an OAuth Server-to-Server technical account, configured once *inside AEM***. It is **not** authorized by the interactive user's `aio` token, and the `x-content-source-authorization` header is the **BYOM/preview** mechanism (likely not exercised on the native push flow).

**→ Reusing the `aio` user identity for the content read: NO (production); needs-live-test only for a local-dev BYOM preview fallback.**

## Q1 — What token does the AEM-Sites content source accept?

- `fstab.yaml` mountpoint (verbatim, `adobe-rnd/aem-boilerplate-xcom`; `roberttoddhoven/citisignal-one` identical shape):
  ```yaml
  mountpoints:
    /:
      url: "https://author-p130360-e1272151.adobeaemcloud.com/bin/franklin.delivery/adobe-rnd/aem-boilerplate-xcom/main"
      type: "markup"
      suffix: ".html"
  ```
  **Confidence: High** (primary repo source).
- On publish, AEM **pushes** to the Adobe pipeline → EDS publish service → EDS ingests semantic HTML pulled from AEM author via franklin.delivery. **High** ([publishing-from-authoring](https://www.aem.live/docs/publishing-from-authoring); page 403s, via search extract).
- `x-content-source-authorization` exists and forwards a `Bearer <token>` Authorization value **verbatim** to html2md — but it is the generic/**BYOM** path, not the AEM-Sites-specific native mechanism. **High** that the header exists/forwards; **Medium** that it is *not* the native push mechanism ([BYOM](https://www.aem.live/developer/byom), search extract).
- Native read is authorized by a **Site Authentication Token tied to a technical account** (`<id>@techacct.adobe.com`), set in AEM: *Tools → Cloud Services → Edge Delivery Services Configuration → Authentication*. Resolves to an **IMS access token**. **High** ([authentication-setup-authoring](https://www.aem.live/docs/authentication-setup-authoring), [Service credentials](https://experienceleague.adobe.com/en/docs/experience-manager-learn/getting-started-with-aem-headless/authentication/service-credentials)).

## Q2 — Which credential issues the token?

| Credential | Works for content-source read? | Confidence |
|---|---|---|
| **OAuth Server-to-Server** (technical account, Developer Console) | **Yes — intended.** Auto-generated read-only technical account per AEM author env; member of a Product Profile granting READ. | High ([generating-access-tokens](https://experienceleague.adobe.com/en/docs/experience-manager-cloud-service/content/implementing/developing/generating-access-tokens-for-server-side-apis)) |
| **User IMS token** (`aio login`) | Used by **Universal Editor write-back**, not the EDS publish read. | High (UE path); Medium (that it's *not* accepted for EDS read) |
| **Local dev access token** (24h) | **Local development only**; per-env, ≤24h. | High ([Local Dev Access Token](https://experienceleague.adobe.com/en/docs/experience-manager-learn/getting-started-with-aem-headless/authentication/local-development-access-token)) |

**Scopes/profiles:** technical account must be in an AEM **Product Profile** granting **READ** on the content tree; generated by an IMS org admin. Exact scope strings not enumerated in fetchable docs — **live-test item**.

## Q3 — Reuse the existing `aio` user identity? **No / Needs-live-test**

1. Native flow authorizes reads with an **environment-specific technical-account credential configured server-side in AEM** — a read-only service identity, "the same for all sites on a single AEM author environment." Not the end-user's identity. **High.**
2. The user's `aio` token is the **Universal Editor** editing/write-back leg — a different system. No documented path where the EDS publish pipeline borrows it. **Medium-High** (inferred from clean separation).
3. The only place a user-supplied Bearer is accepted verbatim is **BYOM preview**; whether a *user* IMS token (vs technical-account) is accepted by franklin.delivery with read scope is **the precise live-test question**. **Low** that it works.

## Q4 — JWT vs OAuth S2S

OAuth Server-to-Server is the **only** current standard. JWT Service Account: no new creds after 2024-06-03; **EOL 2025-06-30**; auto-converted by ~2026-03-01. Today (2026-06-11) **JWT is fully dead.** **High** ([migration guide](https://developer.adobe.com/developer-console/docs/guides/authentication/ServerToServerAuthentication/migration), [JWT deprecation](https://experienceleague.adobe.com/en/docs/experience-manager-65/content/security/jwt-credentials-deprecation-in-adobe-developer-console)).

## Q5 — Cloud Manager (boundary)

Distinct credential scoped to **Cloud Manager roles** (Deployment Manager, etc.), orthogonal to the AEM author **Product Profile (READ)** that authorizes content read. Different product context, different role/profile. **High.** Keep in Slice 3 as scoped.

---

## Recommendation

- **Credential for content-source read:** an **OAuth Server-to-Server** technical account that is a member of an AEM **Product Profile granting READ** — i.e., the identity AEM wires into the EDS "Site Authentication Token" config. For the **native push flow this is configured inside AEM**, so the extension likely passes **no** content-read token (`getContentSourceAuthorization()` → `null`) and instead **guides + verifies** the in-AEM EDS auth setup.
- **Reuse `aio` identity: No** for production; needs-live-test only for a local BYOM preview fallback. Do **not** design the happy path around silent reuse.
