# Research: Unify the two mesh-deploy code paths

**Date:** 2026-06-18
**Backlog:** `.rptc/backlog/unify-mesh-deploy-pipeline.md`
**Goal:** Collapse the dashboard command's inline `aio api-mesh:update` into the shared
`deployMeshComponent` service so there's one deploy core.

## The two implementations

### `deployMeshComponent` (service) — `mesh/services/meshDeployment.ts`
Used by **creation** (`meshSetupService`, `createProject`) and **both resets**.
1. **Builds** the mesh component if it has a build script (`npm install` + `npm run build`) →
   regenerates `mesh.json` from `mesh.config.js` (no-ops when there's no build script).
2. Validates `mesh.json`.
3. **create-or-update**: `aio api-mesh:create` vs `:update` chosen from `existingMeshId`.
4. `waitForMeshDeployment` verification.
5. Returns `{ success, data: { meshId, endpoint } }`; reports via `onProgress(msg, subMsg)`.
6. **No** persistence, **no** lock, **no** dashboard coupling (callers own those).

### `DeployMeshCommand` (inline) — `mesh/commands/deployMesh.ts`
The dashboard "Deploy Mesh" button.
1. **No build step** — reads and deploys the existing `mesh.json` as-is.
2. Hardcoded `aio api-mesh:update` (**never** create).
3. Rich dashboard telegraphing: `sendMeshStatusUpdate('deploying', …)` per phase +
   `withProgress` notification + verbose `[1/3]` logs.
4. `waitForMeshDeployment` verification (same util).
5. **Persists**: `meshComponent.status/metadata.meshId`, `updateMeshState`,
   `meshStatusSummary='deployed'`, `saveProject`, final `sendMeshStatusUpdate('deployed', endpoint)`,
   success toast, `meshActionTaken` flag.
6. Wrapped in an `ExecutionLock`; on failure sets error status + View-Logs toast.

## What unification looks like

Make the command a thin orchestrator that keeps its **lock + pre-flight + dashboard
telegraphing + persistence**, but delegates **build + deploy + verify** to
`deployMeshComponent`:
- Acquire lock, run pre-flight (`ensureProjectAdobeContext`) + App Builder gate (already there).
- Fetch existing mesh id (`fetchMeshInfoFromAdobeIO`, as the reset flow already does).
- Call `deployMeshComponent(meshPath, executor, logger, onProgress, existingMeshId)` with an
  `onProgress` that drives `sendMeshStatusUpdate` + the `withProgress` messages.
- On success, run the existing persistence; on failure, the existing error UX.

Net: one place issues `aio api-mesh:*`; the command keeps only orchestration/UX.

## Behavior changes this introduces (the real decisions)

1. **Build on dashboard redeploy.** Today the dashboard deploy does **not** build — it ships
   the existing `mesh.json`. The service **does** build (when a build script exists). Since the
   redeploy is usually triggered by a "Redeploy mesh to apply changes" prompt after a config
   edit, building is arguably **more correct** (a mesh.config.js change actually takes effect),
   and the service already no-ops when there's no build script. But it's slower and can surface
   build failures that the current path never hits. → **PM decision.**
2. **create-or-update vs update-only.** The command hardcodes `update`; the service picks
   create/update from the fetched mesh id. Switching is strictly more robust (handles a
   not-yet-deployed mesh) and matches the reset flow. → low risk, recommend adopt.
3. **Status sub-message wording.** The service's `onProgress` sub-messages differ slightly from
   the command's current phrasings; mappable but not byte-identical. → cosmetic.

## Constraints (carry over from the gate work)
- The deploy primitive stays **non-interactive** (creation calls it; no prompts inside).
- Don't regress the resets' `withOrgContext` per-invocation env targeting.
- Preserve the command's persistence + dashboard status contract exactly (card grid, badges).

## Recommendation
Proceed with the delegation. Adopt create-or-update (#2). For the build step (#1), recommend
**including** the build (correctness; service already no-ops without a build script) — but this
is the one user-visible behavior change, so confirm with the PM before implementing. #3 is
cosmetic; keep the dashboard-facing wording the command already uses where it matters.

## User-flow analysis (the two decisions, grounded in real flows)

How `mesh.json` is produced and mutated, traced through the actual flows:

- **`mesh.json` is a build artifact, not a static file.** It's generated by the component's
  `npm run build` (`scripts/build-mesh.js`) from `mesh.config.js` + schema/resolvers, which
  bakes in the current `.env` values. (`meshDeployment.ts:19-24,67-76`)
- **Creation builds + deploys.** Creation runs `deployMeshComponent` → builds then deploys, so
  after a successful creation the mesh **exists** (has a `meshId`). (`meshSetupService.ts`)
- **Both resets build + deploy** via `deployMeshComponent` with an `existingMeshId`. They are
  the model of correctness.
- **Configure save writes only `.env`** — `regenerateEnvFiles` (`configure.ts:546-571`) writes
  root + component `.env` and **does not** rebuild `mesh.json` or call the build. It sets
  `meshStatusSummary = 'stale'` and shows "Redeploy mesh to apply changes".

### Decision (a) — build before deploy: REQUIRED, not just "nicer"
The canonical "Redeploy mesh to apply changes" flow is: Configure writes new `.env` → marks
stale → user clicks **Redeploy Mesh** → `deployMesh` command. But the command **deploys the
on-disk `mesh.json` as-is with no build** — and that `mesh.json` still has the **old** values
baked in (Configure didn't rebuild it). **So the current dashboard redeploy ships stale config —
the change the user just made does not take effect.** The build step regenerates `mesh.json`
from the new `.env`, which is exactly why creation and both resets always build first. Building
isn't a behavior *risk* here — it's the fix for a latent correctness bug, and it makes the
dashboard path consistent with every other deploy path.

### Decision (b) — create-or-update: handles a real recovery flow
After a *successful* creation the mesh exists, so the everyday redeploy is `update`. But the
dashboard deploy is reachable with **no deployed mesh**: a creation whose mesh-deploy phase
failed (component cloned, no `meshId`), or a meshless→mesh transition. The command's hardcoded
`update` (`deployMesh.ts:187`) **fails** in those cases. The service's `existingMeshId`-driven
create-or-update recovers them. Source the id from **`fetchMeshInfoFromAdobeIO`** (the remote
truth the headless reset already uses) rather than stale local `metadata.meshId`, so a
remotely-deleted or remotely-existing mesh is handled correctly.

### Conclusion
Both choices are confirmed by the flows — and the unification is not merely a refactor: it
**fixes two latent bugs** on the dashboard path (stale redeploy; can't create a never-deployed
mesh). Delegating the command to `deployMeshComponent` resolves both by construction, since that
primitive already builds and chooses create/update. Cosmetic status wording (#3) stays as the
command phrases it via the `onProgress` adapter.
